## 0. Глосарій

| Термін | Розгортка | Значення в проєкті |
|---|---|---|
| **Surface** | поверхня частинки | центральна абстракція; усі Pydantic-моделі обертаються навколо неї |
| **Ligand** | прищеплена молекула | SMILES + точка прищеплення + орієнтація |
| **Grafting density** | щільність прищеплення | `ligands per nm²` або частка зайнятих силанолів |
| **Janus** | двосторонній | анізотропна функціоналізація (одна півсфера ≠ інша) |
| **Bifunctional** | двофункціональний | дві хімічно різні системи лігандів на одній сфері |
| **Orthogonal func.** | ортогональна | лігандні системи не реагують одна з одною (HMDS+GLYMO) |
| **MLP** | ML potential | Machine-Learning Interatomic Potential (MACE, NequIP, Allegro) |
| **MD / NEMD** | (Non-)Equilibrium MD | стандартна / зсувна молекулярна динаміка |
| **PMF** | Potential of Mean Force | вільна енергія вздовж колективної координати |
| **SASA** | Solvent Accessible Surface Area | площа поверхні, доступна розчиннику |
| **HTS** | High-Throughput Screening | масовий скринінг (Tier 0/1) |
| **Tier 0/1/2** | three-tier pipeline | дешеве cheminformatics → MLP-MD → DFT |
| **AL** | Active Learning | автоматичний цикл донавчання MLP при виявленні невідомих конформацій (uncertainty-driven) |
| **UQ** | Uncertainty Quantification | оцінка впевненості MLP у передбаченні сил/енергій (committee, ensembles, evidential) |
| **GCMC** | Grand Canonical Monte Carlo | метод симуляції адсорбції газів за фіксованого μ (для UC-5 CO₂) |
| **InChIKey** | hashed molecular identifier | 27-символьний хеш SMILES для ключа Molecule Cache |
| **Provenance** | походження даних | "measured" vs "model_based" vs "literature" — обов'язкове поле |
| **UC-N** | Use-Case N | конкретна наукова задача (UC-1 = HBM4) |
| **DAG** | Directed Acyclic Graph | граф залежностей між задачами Prefect |
| **Wave** | реліз-хвиля | UC-кошик однієї версії (Wave 1 → v0.1, Wave 2 → v0.5, Wave 3 → v1.0) |
| **Cardboard prototype** | картонний макет | UI/звіт з фейковими даними для UX-валідації від PI до того, як написана логіка |
| **Single-Node, Single-GPU** | архітектурне обмеження v0.1 | усі задачі планувальника працюють у припущенні «один host, один GPU» |
| **Hybrid pipeline** | Screening Flow + Deep-Study Flow | дві ортогональні pipeline-форми, які платформа підтримує |

## 1. Vision & Problem Statement

### 1.1 Що це і навіщо (для гранту, 1 параграф)

**Si nano Factory** — це доменно-специфічна in-silico платформа для
систематичного дизайну функціоналізованих кремнеземних наночастинок з
прогнозованими поверхневими властивостями. Платформа замикає експериментальний
цикл «синтез → характеризація → ітерація» в обчислювальний цикл «генерація →
скринінг → ML-MD → quantum validation → синтез-протокол» і знижує час
ітерації від тижнів (wet lab) до годин (in silico). Це перший в Україні і
Словаччині «цифровий двійник» поверхневої хімії силікатів, побудований на
однієї робочої станції замість дата-центру, і покриває білу пляму foundation-
model-лабораторій (DeepMind, Microsoft) — *аморфні гібридні органо-неорганічні
поверхні з керованою функціоналізацією*.

### 1.2 Кінцеві користувачі

### 1.3 KPI на 12 місяців

| KPI (primary) | Target | Чому це primary |
|---|---|---|
| **Search Space Reduction** | **≥99 %** (з 10 000 кандидатів → ТОП-3 для wet-lab) | Це і є value proposition платформи; wet-lab кількість мізерна, а скринінг — масовий |
| Use-cases у pipeline | ≥3 (UC-1 done; +UC-3 Heavy metals; +UC-2 REE) | Демонструє доменну ширину |
| Deep-Study повних PoC / тиждень | ≥5 (на 1 × RTX 3090) | Throughput, прив'язаний до hardware |
| Screening cumulative ligand candidates | ≥10 000 | Tier-0 demo |
| Активних MLP-моделей у model-zoo | ≥5 | Reuse / fine-tuning база |
| Q1-публікацій (submitted) | ≥1 (паpler-paper) | Академічна валідація |

| KPI (secondary, bonus) | Target | Why |
|---|---|---|
| GitHub stars after public release | ≥50 | Community proxy |
| Wet-lab handoff completed (real Stöber) | ≥1 з Melnyk-групою | Закриває "translation gap" |
| HPC-провайдерів інтегровано (див. §15) | ≥1 (LUMI або JURECA) | Майбутній scale-out |

> **Зауваження.** PI правильно зауважив: wet-lab kpi — повільний, не treba робити primary. Тому головна метрика — **Search Space Reduction** (на скільки порядків зменшено простір пошуку для експериментатора). Wet-lab перенесено у secondary.

### 1.4 Hardware constraint (явно)

**v0.1 — Single-Node, Single-GPU.** Усі дизайн-рішення планувальника, бюджету
VRAM/RAM/диску, та паралелізму виходять з припущення «один host, один RTX 3090,
24 GB VRAM, 64 GB RAM, 1 TB sustainable disk». Це **не баг, а умисний design
constraint**: спершу довести цінність на споживчому залізі, потім масштабувати
(див. §15 HPC Integration). Архітектура має дозволяти горизонтальний scale-out
без переписування core-API — лише через зміну backend (див. §15.4 Adapter
pattern).

## 2. Use-Cases (Wave-based release plan)

> **Зміна з v0.1.** PI запропонував згрупувати UC у три «хвилі» за складністю.
> Я приймаю принцип, але **не погоджуюсь з пониженням UC-2 (REE) до Wave 2** —
> аргумент нижче. Решта grouping — як рекомендовано PI.

### Wave 1 — PoC & first-grant story (M0–M6, реліз v0.5)

> Ціль: довести end-to-end платформу + грантова заявка з UC-2 як flagship.
> Технічна простота (моноіони, моноліганди, прості топології).

### UC-1 — HBM4 capillary underfill (BASELINE — DONE)

- **Стан:** PoC завершено `2026-05-01`, артефакти в `prestart/HBM4_underfill_PoC/`.
- **Хімія:** 5–20 нм SiO₂ з 85 % HMDS / 15 % GLYMO у Bisphenol-F смолі.
- **Observables:** shear-thinning curve, PMF, SASA, MACE accuracy.
- **Роль для платформи:** baseline throughput ≈ один повний DFT→MACE→MD цикл за 1 ніч на RTX 3090. Усі чисельні KPI (4 модулі) калібруються по ньому.

### UC-3 — Heavy-metal sorbent (Cu²⁺, Hg²⁺, Pb²⁺ з рідних вод) — **Wave 1**

- **Хімія:** Stöber 100 нм + меркаптопропіл / амінопропіл (MPTMS / APTMS).
- **Топологія:** моноліганди, моноіони з простими [Ne]3d¹⁰ або [Ar]3d⁹ конфігураціями — DFT тривіальний, MACE transferable.
- **Observables:** isotherms (Langmuir / Freundlich константи з MD), kinetics (sticking probability), competitive selectivity.
- **Кінцева рекомендація:** оптимальне MPTMS:APTMS ratio.
- **Чому Wave 1:** найшвидший шлях до wet-lab валідації з Melnyk-групою (експертиза точно є). Хімічно тривіальний, але дає 2-у статтю.

### UC-2 — REE sorbent (Nd³⁺, Eu³⁺, Dy³⁺) — **Wave 1, PI РЕКОМЕНДУВАВ Wave 2**

- **Хімія:** мезопорний SBA-15 + ліганд-хелатор (DTPA-, EDTA-, hydroxypropyl-аміно-силан, або новий з ZINC-screening).
- **Топологія:** лантаноїди вимагають ECP-базисів (Stuttgart-Cologne RSC, def2-TZVPP+ECP) у DFT, обережного спінового handling-у (f-електронні стани). Це **не блокер**, а лише трохи довший Tier-1 (DFT cluster).
- **Observables:** ΔG_bind для кожного REE; селективність ΔΔG (Nd-vs-Eu, Dy-vs-Y); регенерація.
- **Кінцева рекомендація:** *"SBA-15 з 0.3 ligands/nm² ліганда L7, оптимум для Dy"*.
- **Чому я не погоджуюсь з пониженням до Wave 2:**
  1. **Grant story.** Horizon Europe + ERC мають потужну REE-line (EU дуже хоче незалежність від Китаю по рідкісних землях). Жоден сильний грантовий додаток на платформу не пройде без REE-кейсу. Якщо UC-2 у Wave 2 — він готовий вже після того, як перший грант поданий.
  2. **Ризик технологічних деталей менший, ніж здається.** ECP вже стандартизовані в ORCA; PySCF теж їх вміє. Один-два додаткові тижні CPU-часу на DFT-датасет.
  3. **Differentiation з конкурентами.** UC-1 (HBM4) і UC-3 (Cu/Hg) — конкурентоспроможні теми, але не унікальні. UC-2 (REE на функціоналізованому SBA-15 з MLP-AL) — це white space.
- **Компроміс:** залишаємо UC-2 у Wave 1 з прозорою позначкою «expensive Tier-1». Якщо до M5 видно перевитрату GPU-годин — переносимо на Wave 2.

### Wave 2 — Complex Interfaces (M6–M9, реліз v1.0)

> Ціль: розширення на складніші топології, демонстрація matrix effects.

### UC-4 — Mesoporous drug carrier (SBA-15 + ibuprofen / curcumin)

- **Хімія:** SBA-15, фосфоновані стінки, контрольоване вивільнення pH-чутливим лігандом.
- **Observables:** loading capacity, release rate as function of pH, ζ-potential proxy.
- **Складність:** великі solvation boxes (~10⁵ молекул води + buffer ions), довші MD-вікна (μs-проміжки) → потребує AL-watchdog для коректності MACE на нових конформаціях.

### Wave 3 — Grand Challenges (M9–M12+, реліз v2.0)

> Ціль: задачі, які потребують **інших методів симуляції** або **HPC scale-out**.

### UC-5 — Amine-grafted CO₂ capture

- **Хімія:** APTES / PEI-grafted мезопорний кремнезем, циклічний sorption-regen.
- **Observables:** CO₂ enthalpy of adsorption, water-tolerance, cycling stability.
- **Складність:** правильна термодинаміка адсорбції газу — **GCMC** (а не лише MD).
  Потрібно інтегрувати RASPA / Cassandra поряд з LAMMPS. Це окрема pipeline.

### UC-6 — Janus particle для емульсій

- **Хімія:** один бік TMS, інший — APTES (анізотропна функціоналізація).
- **Observables:** контактний кут, оптимальний баланс HLB, агрегативна стабільність на інтерфейсі.
- **Складність:** сольваційний бокс — **двофазний** (водна + олійна). Розмір 10⁶ атомів, μs-таймскейл → реалістично тільки з **HPC backend** (§15).

### 5.2 Surface-centric Pydantic core (FR-04)

```python
# Це не повний код — тільки концептуальний контракт.
# Повна реалізація — у Module 1 (наступний deliverable).

class NanoparticleSpec(BaseModel):
    """Будь-яка наночастинка в проекті описується ЦИМ об'єктом."""
    core: CoreSpec          # розмір, морфологія, кристалічність
    shell: ShellSpec | None # core-shell? (Fe3O4@SiO2)
    surface: Surface        # ← головна абстракція
    environment: Env        # solvent, T, pH, sorbates

class Surface(BaseModel):
    """The two-dimensional world. Усе живе тут."""
    ligands: list[GraftedLigand]       # кожен ліганд = SMILES + позиція
    distribution: Distribution          # random | janus | patchy | gradient
    grafting_density: float             # ligands / nm²
    coverage_fraction: float            # frac of available silanols
    # Provenance: всі числа мають походження
    _provenance: dict[str, ProvenanceLabel]
```

Surface — це не атрибут частинки, це **обличчя** частинки. Bulk core — boundary
condition, не обчислювана сутність.

### 5.3 Versioning strategy

- **Code:** semantic versioning, `git` + GitHub releases.
- **Data (datasets, trajectories):** DVC remote → MinIO bucket `factory-dvc/`.
- **Models (MACE checkpoints):** MLflow Model Registry stages
  (`Staging` / `Production` / `Archived`).
- **Specs (NanoparticleSpec snapshots):** stored as JSON in PostgreSQL
  з `spec_hash` (SHA-256 канонізованого JSON) як primary key.

### 5.4 Molecule Cache (PI recommendation, accepted)

Будь-який *expensive single-point розрахунок* — DFT energy, xTB descriptor,
RDKit-derived feature — має йти через єдиний кеш-шар з **InChIKey + level-of-
theory + parameter-set** як композитним ключем.

```python
# Контракт — деталі в Module 1.
class MoleculeCache:
    def get_or_compute(
        self,
        molecule: Molecule,                  # SMILES → canonical → InChIKey
        method: ComputationalMethod,         # ('B3LYP', 'def2-SVP', 'D3BJ', ...)
        compute_fn: Callable[[], MoleculeResult],
    ) -> MoleculeResult: ...
```

**Storage layout:**
```
PostgreSQL table: molecule_cache
  PK:    (inchikey, method_hash)
  cols:  energy_hartree, forces_blob, dipole, homo, lumo,
         charges, computed_at, computed_with_version, provenance
  INDEX: (inchikey), (method_hash), (computed_at)
```

**Why це критично:**
1. Аміно-пропіл-силан фігурує у UC-3, UC-4, UC-5 — рахувати його раз, не тричі.
2. Tier-0 screening вибирає 100 кандидатів з 10 000; багато повторюваних
   substructure-оточень — кеш буде влучати на 30-50 %.
3. Перевикористання між UC-ами знижує загальний DFT budget у 2-3×.

**Invalidation:** ніколи. DFT energy за фіксованою методикою — детермінована.
Зміна методики → новий `method_hash` → новий запис, старий лишається.

## 6. Computational Architecture

### 6.1 Two orthogonal pipeline shapes (revised after PI feedback)

> **Зміна з v0.1.** PI запропонував tier-визначення, в яких DFT (Tier 1) йде
> ПЕРЕД MLP-MD (Tier 2). Я погоджуюсь, але фіксую, що **це тільки одна з двох
> pipeline-форм**. HBM4 PoC, наприклад, працював по іншій логіці. Тому в v0.2
> архітектура чітко розділяє:

#### A. Screening Flow — «знайти кращого з багатьох»

> Вхід: пул лігандів-кандидатів (10² – 10⁵). Вихід: ТОП-3 для wet-lab.
> Ідеологія PI: Tier 0 → Tier 1 → Tier 2.

```
┌───────────────────────────────────────────────────────────────────────┐
│  S-Tier 0 — CHEMINFORMATICS                  ~10⁴ candidates / day   │
│  CPU pool (i9, 24 threads), seconds per molecule                      │
│  • RDKit: SMILES parse, 2D descriptors, SA-score, Lipinski-like       │
│  • SASA estimate (probe sphere on UFF/MMFF geometry)                  │
│  • xTB / MOPAC: HOMO-LUMO, dipole, partial charges                    │
│  → ranked CSV (top ~1 000)                                            │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
┌───────────────────────────────────────────────────────────────────────┐
│  S-Tier 1 — QUANTUM CLUSTER BINDING          ~10³ candidates / week  │
│  CPU (ORCA) + cautiously GPU (GPU4PySCF), hours per molecule          │
│  • Build minimal "ligand–metal–support" cluster (~30-80 atoms)        │
│  • Compute ΔE_bind at B3LYP-D3BJ/def2-TZVP (or RSC-ECP for REE)       │
│  • Cache via InChIKey (§5.4) — друга поява = безкоштовно              │
│  → ranked list (top ~50)                                              │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
┌───────────────────────────────────────────────────────────────────────┐
│  S-Tier 2 — MLP-MD + PLUMED                  ~5-50 candidates / week │
│  GPU (RTX 3090), days per candidate                                   │
│  • Use MACE-MP-0 fine-tuned (or domain MACE if Deep-Study вже є)      │
│  • Solvated MD з explicit ions, 1-10 ns                               │
│  • PLUMED: PMF along ligand–metal distance → ΔG_bind in solution      │
│  • Active-Learning Watchdog (FR-32) ловить unphysical конформації     │
│  → ТОП-3 кандидатів + повний звіт                                     │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
              [SynthesisRecipe → Wet-Lab handoff]
```

#### B. Deep-Study Flow — «глибоке дослідження одного»

> Вхід: один обраний `NanoparticleSpec`. Вихід: повноцінна стаття + reusable
> MACE-модель для свого хімічного класу. Це і є форма HBM4 PoC.

```
┌───────────────────────────────────────────────────────────────────────┐
│  D-Tier 0 — Build & Pack                     ~minutes                │
│  CPU; topology + Packmol solvation                                    │
│  • NanoparticleSpec → ASE.Atoms + topology.json                       │
│  • Add solvent (Bisphenol-F, water+ions, ...) via Packmol             │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
┌───────────────────────────────────────────────────────────────────────┐
│  D-Tier 1 — DFT Active-Learning dataset      ~hours-days             │
│  CPU (ORCA) + GPU (GPU4PySCF), 200-500 frames                         │
│  • Initial random rattling around equilibrium                         │
│  • Iterative: MACE-MD → UQ-flag → DFT → retrain                       │
│  → real_quantum_dataset.extxyz (cache hits via InChIKey)              │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
┌───────────────────────────────────────────────────────────────────────┐
│  D-Tier 2 — MACE training                    ~hours                  │
│  GPU exclusive lock                                                   │
│  • 200-500 epochs, SWA, hyperparams from registry                     │
│  → domain MLP checkpoint, MLflow registry "Staging"                   │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
┌───────────────────────────────────────────────────────────────────────┐
│  D-Tier 3 — Production MD                    ~hours-days             │
│  GPU long-hold lock                                                   │
│  • 5-50 nm particle, 100-1000 ps Langevin/NPT                         │
│  → trajectory in .zarr (NFR-14 compression)                           │
└───────────────────────────────────────────────────────────────────────┘
                            ↓
┌───────────────────────────────────────────────────────────────────────┐
│  D-Tier 4 — Free Energy + Analysis           ~hours-days             │
│  GPU; PLUMED umbrella / metadynamics                                  │
│  • PMF, RDF, SASA, viscosity-proxy, ΔG_bind                           │
│  → SynthesisRecipe + статті-чорновик                                  │
└───────────────────────────────────────────────────────────────────────┘
```

#### C. How the two flows interact

| Випадок | Workflow |
|---|---|
| Новий клас хімії, ще немає MLP | Deep-Study (D-Tier 0–4) → новий MACE → потім доступний для Screening |
| Скринінг лігандів у відомому класі | Screening (S-Tier 0–2) — використовує існуючий MACE з Deep-Study |
| Один кандидат пройшов скринінг → треба деталі | Screening вихід → запуск Deep-Study на топ-1 |
| HBM4 PoC | Чистий Deep-Study (вже виконаний) |
| UC-3 first attempt | Screening (бо є foundation MACE-MP-0) → Deep-Study на топ-1 |

**Правило великого пальця:** **Screening для широкого пошуку, Deep-Study для
narrative-paper.** Жоден не дублює іншого.

### 6.3 GPU lock + VRAM budget

```python
class GPULock:
    """Singleton. Лише одна задача тримає GPU за раз.
    Декларація min_vram_gb перед acquire; OOM → автоматичний fallback."""

    async def acquire(self, min_vram_gb: float, task_id: str): ...
    async def release(self, task_id: str): ...

class GPUBudget(BaseModel):
    available_vram_gb: float = 22.0   # 24 - 2 reserved для драйвера
    current_holder: str | None = None
    queue_depth: int = 0
```

CPU-задачі не блокують GPU; GPU-задачі НЕ запускаються паралельно з іншими
GPU-задачами. Виняток: легкий MACE-inference (≤2 GB VRAM) може co-locate з
CPU-only DFT (за умови `cuda:0` shared mode).

### 6.4 Cold/hot/long-hold queue lanes

| Lane | Тип задач | Час життя | Concurrency |
|---|---|---|---|
| **cold** | RDKit, SA-score, packing | <1 хв | до 24 (CPU pool) |
| **warm** | DFT single-point, xTB | 1-30 хв | до 4 (CPU + memory) |
| **hot** | MACE-train, short MLP-MD (Tier 1) | 1-3 год | 1 (GPU lock) |
| **long-hold** | Production MD (500 ps), umbrella sampling | 6-24 год | 1 (GPU lock, exclusive) |

## 8. Module-level breakdown (mapping → Master Prompt)

### 8.1 Module 1 — Universal Parametric Engine (`sinanofactory/core/`)

- `core/spec.py` — `NanoparticleSpec`, `CoreSpec`, `ShellSpec`, `Surface`,
  `GraftedLigand`, `Env`, `ProvenanceLabel`.
- `core/builder.py` — `build_topology(spec) → ASE.Atoms + topology.json`.
  Підтримує Stöber-сферу, SBA-15-pore, core-shell.
- `core/distributions.py` — Fibonacci sphere, Janus split, patchy patches,
  random Poisson.
- `core/hash.py` — канонізований SHA-256 для cache keys.
- `core/registry.py` — реєстр відомих ядер та шарів (factory pattern).
- `tests/test_spec.py` — property-based tests (Hypothesis).

### 8.2 Module 2 — Ligand & Precursor Screening (`sinanofactory/screening/`)

- `screening/sources/` — connectors: `pubchem.py`, `zinc.py`, `chembl.py`,
  `local_csv.py`. Кожен має кеш на диску (24 год TTL за замовчанням).
- `screening/filters/` — `sa_score.py`, `lipinski.py`, `sasa_fit.py`,
  `electronic.py` (xTB / MOPAC wrapper).
- `screening/pipeline.py` — Prefect flow: source → filter chain → ranked CSV.
- `screening/silane_specific.py` — спеціалізація для R-Si(OR')₃: гарантує
  тригідрол-голову, тестує fluorinated tail, REE-chelator pattern,
  pH-responsive moiety.

### 8.3 Module 3 — Infrastructure Scheduling (`sinanofactory/orchestrator/`)

- `orchestrator/gpu_lock.py` — асинхронний GPU lock (з file-lock fallback).
- `orchestrator/budget.py` — VRAM/RAM budget tracker.
- `orchestrator/lanes.py` — cold / warm / hot / long-hold queue lanes.
- `orchestrator/flows/` — Prefect flows для UC-1…UC-N (один flow на UC).
- `orchestrator/recovery.py` — OOM detection + retry logic.
- `orchestrator/cli.py` — Typer entrypoint (`factory run`, `factory queue`,
  `factory replay`).

### 8.4 Module 4 — In Silico → Wet Lab Translator (`sinanofactory/translation/`)

- `translation/observables.py` — registry усіх observables (RDF, PMF, SASA,
  ΔG_bind, isotherm, viscosity-proxy) + how to compute з трае.
- `translation/recipe.py` — `SynthesisRecipe(BaseModel)`: list реагентів,
  стехіометрія, температурний профіль, час, очищення.
- `translation/render.py` — `recipe → Markdown` + `recipe → PDF` (через
  WeasyPrint або Pandoc).
- `translation/justify.py` — automatic "why" paragraph: "обрано L7 з
  ratio 0.4:0.6, бо ΔG_bind = -X kJ/mol, q95 SASA accessibility = Y %".
- `translation/templates/` — Jinja2 шаблони для трьох рівнів детальності
  (one-pager, full report, statt'я-чорновик).

### R-06 refinement: чому **time-based release** замість dual-licensing

> PI запропонував dual-license (Apache-2.0 на core + private MACE weights / recipes для патенту). Я погоджуюсь з духом, але пропоную простішу імплементацію.

**Проблема dual-licensing:** різні файли в одному репо мають різні ліцензії →
`SPDX-License-Identifier` headers, `LICENSE.commercial`, NOTICE-файли,
contributor agreements (CLA). Юридичний оверхед, який для команди з 1-2 осіб
з'їдає 5-10% часу. Високий ризик помилок.

**Простіша альтернатива — time-based release:**

| Артефакт | Default ліцензія | Затримка | Чим контролюємо |
|---|---|---|---|
| Core code (`sinanofactory/`) | Apache-2.0 з дня 1 | none | public GitHub з M0 |
| UC-specific flow code | Apache-2.0 з дня 1 | none | той же репо |
| Загальні MACE-моделі (water, silica, Bisphenol-F) | Apache-2.0 + CC-BY-4.0 | при release `v0.5` | Hugging Face / Zenodo |
| **Конкретні "winning recipes"** (e.g., specific REE-ligand combo) | Patent disclosure → CC-BY-4.0 | **6-12 міс. після discovery** | Private Postgres tag `IP_PROTECTED=true` до подачі патенту, потім публікація |
| **Спецналаштовані MACE-ваги для патентованих рецептів** | те ж | те ж | Same flag |

**Implementation:** один булевий флаг у `MoleculeCache.row.ip_protected`. Якщо
`true` — `factory export --public` пропускає його. Все інше працює як єдиний
open-source. Жодних license headers per-file.

**Discovery → patent decision tree** (хто вирішує):
```
Tier-2 result з ΔG_bind > поріг + новизна > 80 % у literature search
                     │
                     ▼
        [hot-flag в MLflow run]
                     │
                     ▼
        [PI рев'ю протягом 5 робочих днів]
                     │
              ┌──────┴──────┐
              ▼             ▼
       File patent     Ні → release CC-BY
       (12 міс. lock)
```

## 14. HPC Integration Strategy (NEW v0.2)

> **Запит PI.** «Передбач для розробки HPC-інтеграцію з іншими суперкомп'ютерами
> чи постачальниками апаратних потужностей.» Цей розділ — повна стратегія
> від single-node v0.1 до multi-cluster v2.0.

### 14.1 Capability ladder (4 рівні зрілості)

| Рівень | Версія | Backend | Що дозволяє |
|---|---|---|---|
| **L0** | v0.1 (M3) | local single-GPU | один Deep-Study run / day, як HBM4 PoC |
| **L1** | v0.5 (M6) | local + 1 SLURM кластер | overflow при busy GPU; нічні batch S-Tier 1 |
| **L2** | v1.0 (M9) | local + SLURM + 1 cloud (AWS/GCP) | elastic burst для grant-deadlines; Tier-2 ансамблі |
| **L3** | v2.0 (M12+) | multi-cluster federated | UC-6 Janus 10⁶-атомні MD; UC-5 GCMC sweeps; multi-PI |

**Ключове правило L0→L3.** Користувацький код (`NanoparticleSpec` + Prefect
flow) **не змінюється**. Перемикання — лише `--backend=...` або змінна
середовища `FACTORY_BACKEND=lumi`.

### 14.2 Backend Adapter pattern (FR-36, NFR-16)

```python
# Контракт. Повна реалізація в Module 3.

class BackendAdapter(Protocol):
    """Один інтерфейс для всіх обчислювальних бекендів."""

    name: str                          # "local", "slurm-jurec", "aws-batch", "lumi-mi250"
    capabilities: BackendCapabilities  # max_vram_gb, has_mpi, has_internet, ...

    async def submit(self, task: Task) -> RemoteJobHandle: ...
    async def stage_in(self, files: list[Path]) -> RemoteFileSet: ...
    async def stage_out(self, handle: RemoteJobHandle, dest: Path) -> None: ...
    async def status(self, handle: RemoteJobHandle) -> JobStatus: ...
    async def cancel(self, handle: RemoteJobHandle) -> None: ...
    def cost_estimate(self, task: Task) -> CostUSD: ...
```

### 14.6 Multi-backend scheduling logic (v1.0+)

```python
def choose_backend(task: Task, available: list[BackendAdapter]) -> BackendAdapter:
    """Спрощене rule-based рішення (v1.0).
    v2.0 буде RL-based або cost-aware MILP."""

    # Hard requirements first
    feasible = [b for b in available if b.can_run(task)]
    if not feasible:
        raise NoFeasibleBackend(task)

    # Latency-sensitive (cardboard prototype, debug runs) → local
    if task.priority == "interactive":
        return next((b for b in feasible if b.name == "local"), feasible[0])

    # Long-hold + GPU-exclusive (D-Tier 3 production MD) → academic HPC (free)
    if task.estimated_hours > 12 and task.requires_gpu_exclusive:
        return next((b for b in feasible if b.kind == "academic_hpc"), ...)

    # Embarrassingly parallel (S-Tier 1 ORCA) → cloud spot (cheap, elastic)
    if task.is_embarrassingly_parallel:
        return cheapest(feasible, kind="cloud_spot")

    # Default → local if free, else cheapest
    if local := next((b for b in feasible if b.name == "local" and b.is_free()), None):
        return local
    return cheapest(feasible)
```
